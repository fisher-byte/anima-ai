# 决策模块卡死修复与 UI 重设计

*日期: 2026-03-18 | 版本: v0.5.30 | 作者: fishbeta*

---

## 背景

从 v0.5.27 引入决策闭环 UI（DecisionCard、LingSiTracePanel）以来，陆续收到两类严重卡死反馈：

1. **点击「查看轨迹」或「决策依据」后页面完全无响应**，需要强制刷新才能恢复
2. **点击「采纳建议」后弹窗打开极慢**，有时超过 10 秒，期间 UI 完全冻结

两个问题表面不同，根源都指向同一件事：**主线程在 React rendering 或同步 I/O 中被完全占满**。

---

## 问题一：查看轨迹/决策依据点击卡死

### 问题描述

在 streaming 回答过程中（或 streaming 刚结束时），点击 LingSiTracePanel 的「查看轨迹」按钮，页面立即变得无响应。有时能在 1–2 秒后自动恢复，有时需要手动刷新。

复现路径：
1. 发起一次 LingSi 决策问题
2. 回答开始 streaming
3. streaming 期间或刚结束后，点击回答底部的「决策依据」展开面板
4. 点击面板内「查看轨迹」按钮
5. → 页面卡死

### 根因分析

**Portal 嵌在高频渲染树内部**

原始实现（v0.5.29 及之前）：

```
AnswerModal
  └── turns.map(turn => ...)         ← 每个 SSE token 都触发 re-render
        └── LingSiTracePanel          ← 每次都跟着 re-render
              └── createPortal(       ← Portal 在循环内创建
                    <LingSiTraceModal ...>,
                    document.body
                  )
```

`LingSiTracePanel` 内部管理完整的 Portal 生命周期（含 `document.body.overflow = 'hidden'` 的副作用）。streaming 期间每个 token 都会触发 `turns` 数组更新，进而触发整棵 turns 渲染树的 VDOM diff，包括每个 turn 下面的 `LingSiTracePanel` 及其 Portal 子树。

当用户点击「查看轨迹」时，React 的 `setState` 更新与持续的 streaming token 更新同时争抢调度，主线程被大量同步 diff 工作饱和，事件循环无法及时响应用户输入，表现为卡死。

**两个叠加因素**：
- streaming 时 token 频率可达每秒 20–30 个，每个 token 都是一次 React 调度
- Portal 的 `document.body.overflow` 副作用在每次 render 都会检查/设置，产生额外的 DOM 操作

### 解决方案：架构提升 Portal 到渲染树顶层

将 `LingSiTraceModal` 从 `LingSiTracePanel`（turns 循环内）提升到 `AnswerModal` 根层：

```
AnswerModal                            ← 持有 traceData state
  ├── turns.map(turn => ...)
  │     └── LingSiTracePanel           ← 不再持有 Portal
  │           └── [onOpenTrace cb]     ← 仅通过回调上报
  └── {traceData && <LingSiTraceModal  ← Portal 只在顶层渲染一次
            data={traceData}
            onClose={() => setTraceData(null)}
          />}
```

关键改动：
- `LingSiTracePanel` 新增 `onOpenTrace?: (data: TraceData) => void` prop，点击时触发回调，自身不再持有任何 Portal 状态
- `AnswerModal` 在根层持有 `traceData: TraceData | null` state；streaming 期间该 state 为 null，不存在任何 Portal
- `LingSiTraceModal` 成为独立顶层组件，完全脱离 turns 渲染循环

效果：streaming 期间 turns 树的高频更新不再波及 Modal/Portal 的任何逻辑。

### 附加修复：streaming 期间禁用触发按钮

即使架构提升后，在 streaming 进行中触发 expand/collapse 或 trace-view 仍可能引发并发 state 更新。在 `isStreaming` 时将这些按钮设为 `disabled`，作为防御性措施。

## 问题二：openModalById 超大 JSONL 同步阻塞

### 问题描述

点击「采纳建议」或从 DecisionHub 点击一条进行中决策时，弹窗打开极慢甚至卡死。

复现路径：
1. 项目使用一段时间，`conversations.jsonl` 积累到较大体积（数 MB 级）
2. 从 DecisionHub 点击一条进行中决策
3. 调用 `openModalById(id)` 查找对应对话
4. → 页面冻结数秒

### 根因分析

`openModalById` 的原始实现：

```typescript
const raw = await storageService.load('conversations.jsonl')
const lines = raw.split('\n')          // ← 一次性同步切割整个文件
for (const line of lines) {
  const conv = JSON.parse(line)
  if (conv.id === id) { found = conv; break }
  if (scanned++ > scanLimit) break     // ← 这个 guard 其实是 dead code（见后）
}
```

`split('\n')` 对整个 JSONL 字符串（可能几 MB）做同步操作，结果是一个包含数千元素的数组。这个操作本身不慢，但 `JSON.parse` 对每一行的循环调用在主线程是完全同步的，无法被中断，导致 UI 冻结。

另外，目标对话通常是最近的记录，在文件末尾；从头遍历反而是最差情况。

### 解决方案：iterLinesFromEnd generator + cooperative yield

```typescript
async function* iterLinesFromEnd(raw: string, limit: number) {
  const lines = raw.split('\n').filter(Boolean).reverse()
  let count = 0
  for (const line of lines) {
    if (count++ >= limit) return
    yield line
    if (count % 60 === 0) {
      // cooperative yield：让出主线程，避免长时间占用
      await new Promise(r => setTimeout(r, 0))
    }
  }
}

// 调用侧
for await (const line of iterLinesFromEnd(raw, SCAN_LIMIT)) {
  const conv = JSON.parse(line)
  if (conv.id === id) { found = conv; break }
}
```

两个关键点：
1. **从末尾倒序遍历**：最近的对话在末尾，绝大多数情况下只需遍历少量行
2. **每 60 行 cooperative yield**：通过 `setTimeout(r, 0)` 让出一次事件循环，允许 UI 更新和用户输入处理插入，避免长时间独占主线程

### 附：移除 scanLimit dead code

原循环内有一个 `if (scanned++ > scanLimit) break` guard，但 `iterLinesFromEnd` generator 自身已按 `scanLimit` 截断输出，这个 check 在 `for await` 循环里永远不会触发（generator 先 return）。予以清除，避免误导后续维护者。

## 问题三：compareItems 去重逻辑错误

### 问题描述

用户采纳决策并修改回访时间后，在 DecisionHub 面板中看到的回访时间仍是旧值，而不是最新设置的时间。

### 根因分析

`decisionRecords.ts` 的去重逻辑：

```typescript
// 原实现（错误）
const deduped = records.reduce((acc, record) => {
  const existing = acc.find(r => r.id === record.id)
  if (!existing) {
    acc.push(record)
  } else if (compareItems(record, existing) > 0) {  // ← compareItems 按 revisitAt 排序！
    Object.assign(existing, record)
  }
  return acc
}, [])
```

`compareItems` 是为渲染排序设计的，按 `revisitAt`（回访时间）升序排列。用它做去重时，"更大"意味着"回访时间更晚"，而不是"更新"。当用户把回访时间改早了，新记录的 `revisitAt` 反而更小，`compareItems` 判定"旧记录更大"，旧记录被保留。

### 解决方案

去重应当保留"最近更新"的条目，改用 `updatedAt` 比较：

```typescript
} else if (record.updatedAt > existing.updatedAt) {
  Object.assign(existing, record)
}
```

## 问题四：MarkdownLink TypeScript 类型错误

`AnswerModal.tsx` 中为 `react-markdown` 提供的自定义链接组件：

```typescript
// 原实现（报 TS 错误）
const MarkdownLink = ({ href, children }: { href: string; children: ReactNode }) => ...

// 修复：children 改为可选，与 react-markdown components.a 签名对齐
const MarkdownLink = ({ href, children }: { href?: string; children?: ReactNode }) => ...
```

虽然运行时不影响功能，但会导致 `tsc --noEmit` 输出类型错误，干扰 CI typecheck。

## 经验总结

1. **Portal 不应嵌在高频渲染树内**：任何包含 Portal 或 `document.body` 副作用的组件，都应置于渲染树尽量高的位置，或使用 `React.memo` + 稳定引用严格防止不必要的 re-render
2. **大文件解析必须 cooperative**：主线程上的长循环（哪怕是纯 JS，无 I/O）都需要周期性让出事件循环；每 50–100 次迭代插入一次 `await new Promise(r => setTimeout(r, 0))` 是标准做法
3. **排序函数 ≠ 比较函数**：为渲染排序设计的 comparator 通常不能直接复用于去重/合并逻辑，两者的语义不同
4. **streaming 期间的 UI 操作要有防护**：streaming 本身是高频 state 更新，此时触发额外 state 变更容易引发调度问题；对可能并发的操作加 `disabled` 是防御性最佳实践
